Добрый день. Сегодня поговорим о том, что в языке C называют «строками» — и почему это одна из самых коварных и важных тем всего языка.

В Python, Java или C# строка — это объект: у неё есть длина, методы, сборщик мусора. В C строка — это массив символов char, который заканчивается специальным символом '\0'. Всё остальное — следствие этого простого, но жёсткого правила.

Мы пройдём путь от самых основ до тем, которые важны опытным разработчикам: память, lifetime, кодировки, переполнения буфера и проектирование API. В конце у вас будет цельная картина: что такое C-строка, как её использовать и когда от неё лучше отказаться.

Ключевая мысль
В C нет типа string. Есть соглашение: последовательность байтов, завершённая нулевым байтом '\0'.

Часть 1. Для начинающих
1.1. Что такое char
Текст для выступления

Начнём с малого. char — это один символ. Но технически char — это целое число размером один байт. Буква хранится как её код в таблице символов.

char c = 'A';      /* один символ в одинарных кавычках */
char digit = '7';  /* это НЕ число 7, а символ '7' (код 55 в ASCII) */
Запись	Значение
'A'
символ A (код 65)
65
то же число, другое представление
'7'
символ семёрки (код 55), не число 7
1.2. «Строка» — массив char + '\0'
Текст для выступления

Когда мы пишем строку в двойных кавычках, компилятор создаёт массив символов и сам добавляет '\0' в конец. Этот символ не печатается, но именно он говорит функциям: «здесь строка заканчивается».

char hello[] = "Hello";
/* В памяти: 'H' 'e' 'l' 'l' 'o' '\0' */
Главное правило C: строка — последовательность байтов, заканчивающаяся '\0' (нулевой байт, код 0).

1.3. Три способа объявить «строку»
char s1[] = "hi";    /* массив на стеке: 'h', 'i', '\0' */

char buf[10];        /* буфер из 10 байт, содержимое не задано */

Аналогия простая. char arr[] — это ваш блокнот: пишете что хотите. char *ptr = "..." — табличка в музее: смотреть можно, писать поверх нельзя. Попытка изменить литерал через неконстантный указатель — классическая ошибка и неопределённое поведение.

char arr[] = "test";   /* OK: arr[0] = 'T'; */
char *ptr = "test";    /* ОПАСНО: ptr[0] = 'T'; — UB */
1.4. Ввод и вывод строк
#include <stdio.h>
int main(void) {
    char name[50];
    printf("Введите имя: ");
    scanf("%49s", name);   /* лимит: 49 символов + '\0' */
    printf("Привет, %s!\n", name);
    return 0;
}
Важно:

%s в printf и scanf работает с указателем на char, у которого есть '\0'.
scanf("%s", ...) читает до пробела — для строк с пробелами нужен fgets.
Всегда ограничивайте ввод размером буфера: %49s для char name[50].
Более безопасный ввод целой строки:

char line[100];
if (fgets(line, sizeof(line), stdin)) {
  line[strcspn(line, "\n")] = '\0';  /* убрать \n в конце */
}
1.5. Длина строки: strlen
#include <string.h>
char s[] = "Hello";
size_t len = strlen(s);   /* 5 — без учёта '\0' */
strlen идёт по памяти байт за байтом, пока не встретит '\0'.

Если '\0' нет — функция читает чужую память до сбоя. Это неопределённое поведение (UB).

Строка из 5 букв занимает 6 байт в памяти: 5 символов + '\0'.

Часть 2. Для опытных
2.1. Строка — не тип, а конвенция
Текст для выступления

Для опытного разработчика важно сразу снять иллюзию: в C нет встроенного типа «строка». Есть char, массив char[N], указатель char * — и договорённость сообщества использовать нуль-терминированные последовательности байтов.

Это влияет на всё:

функции не знают размер буфера;
компилятор не проверяет границы;
ответственность лежит на программисте и дизайне API.
2.2. Память и срок жизни (lifetime)
char *dangerous(void) {
    char local[] = "oops";
    return local;   /* UB: local уничтожается при выходе */
}
char *ok(void) {
    return "literal";   /* OK: литерал живёт всё время программы */
}

Правило ownership: кто выделил память — тот и освобождает. Если функция возвращает char *, в документации должно быть ясно: это литерал, статический буфер или память, которую должен освободить вызывающий.

2.3. Массив vs указатель — не одно и то же
char a[] = "hi";   /* sizeof(a) == 3 */
char *p = "hi";    /* sizeof(p) == 8 на 64-bit — размер указателя */
В параметрах функции массив деградирует в указатель:

void foo(char s[100]) {
    /* sizeof(s) — размер УКАЗАТЕЛЯ, не 100 */
}
Отсюда классический паттерн:

void safe_copy(char *dst, size_t dst_size, const char *src);
Размер передаётся явно, потому что внутри функции его уже не восстановить.

2.4. const char * — контракт «только чтение»
void print_message(const char *s);
Запись	Что запрещено
const char *p
Менять символы через p
char * const p
Менять сам указатель p
const char * const p
И то, и другое
Хороший API почти всегда принимает входные строки как const char *.

2.5. Кодировки: C-строка ≠ Unicode-текст
C-строка — последовательность байтов, не обязательно «символов» в смысле Unicode:

Кодировка	Особенность
ASCII
1 байт на символ
UTF-8
1–4 байта на code point
locale-зависимые
Поведение ctype.h зависит от окружения
Важно для практики:

strlen считает байты, не графемы;
strcmp, strchr не «понимают» UTF-8 по умолчанию;
разрезать UTF-8 посередине многобайтового символа — ошибка.
Для серьёзной работы с Unicode: ICU, utf8proc, в C23 — char8_t (где поддерживается).

2.6. Почему strcpy и strcat опасны
char buf[5];
strcpy(buf, "Hello");   /* 6 байт с '\0' в буфер на 5 — переполнение */
Это основа класса уязвимостей buffer overflow, из которого выросли тысячи CVE за десятилетия.

Современный подход в C:

char dst[10];
strncpy(dst, src, sizeof(dst) - 1);
dst[sizeof(dst) - 1] = '\0';
Или лучше:

snprintf(dst, sizeof(dst), "%s", src);
Дополнительно: статический анализ, fuzzing, code review, запрет небезопасных функций в политике проекта.

2.7. Строковые литералы
char *s = "Hello" " " "World";   /* компилятор склеивает: "Hello World" */
Соседние литералы компилятор объединяет. Одинаковые литералы могут быть размещены в одной области памяти (string pooling) — но на это нельзя полагаться в переносимом коде.

2.8. Массив символов без '\0' — это не строка
char raw[3] = {'a', 'b', 'c'};   /* нет '\0' */
Такой массив нельзя передавать в strlen или printf("%s") — снова UB.

2.9. Аргументы командной строки
int main(int argc, char *argv[]) {
    for (int i = 0; i < argc; i++) {
        printf("%s\n", argv[i]);
    }
    return 0;
}
argv — массив указателей на C-строки. Это один из самых наглядных примеров, где null-terminated модель встроена в саму платформу.

Несмотря на все проблемы, C-строки до сих пор повсюду — потому что они просты, совместимы с ОС и стали стандартом де-факто для ABI.

Для бинарных данных null-terminated модель не подходит: '\0' внутри данных обрежет строку.

Часть 4. Практические паттерны
4.1. Безопасное копирование
#include <stdio.h>
void copy_string(char *dst, size_t size, const char *src) {
    if (size == 0) return;
    snprintf(dst, size, "%s", src);
}
snprintf гарантирует терминатор, если size > 0.

4.2. Сравнение без учёта регистра (POSIX)
#include <strings.h>
if (strcasecmp(a, b) == 0) {
    /* равны без учёта регистра */
}
На Windows аналог — _stricmp.

4.3. Разбор строки (токенизация)
#include <string.h>
char line[] = "one,two,three";
char *save = NULL;
char *tok = strtok_r(line, ",", &save);
while (tok != NULL) {
    /* обработка tok */
    tok = strtok_r(NULL, ",", &save);
}
Важно:

strtok / strtok_r портят исходную строку, вставляя '\0';
strtok не reentrant — в многопоточности использовать strtok_r;
для разбора без порчи исходника — ручной цикл или своя функция.
4.4. Динамическая строка в куче
#include <stdlib.h>
#include <string.h>
char *dup_string(const char *src) {
    size_t len = strlen(src);
    char *s = malloc(len + 1);
    if (s == NULL) return NULL;
    memcpy(s, src, len + 1);   /* копируем вместе с '\0' */
    return s;
}
/* вызывающий обязан: */
/* free(s); */
4.5. «View» строки без копирования
typedef struct {
    const char *data;
    size_t len;
} string_view;
Плюсы: не нужен '\0' внутри подстроки, можно ссылаться на часть буфера.
Минусы: нужно следить за lifetime исходного буфера; view не владеет памятью.

4.6. Форматирование в буфер
char msg[128];
snprintf(msg, sizeof(msg), "id=%d name=%s", id, name);
Один из самых практичных и относительно безопасных способов собирать строки в C.

C не заставляет вас жить только с char *. Но любой интерфейс с ОС, с Python, с драйвером — почти всегда в итоге сводится к null-terminated строке на границе. Поэтому понимать эту модель обязательно, даже если внутри проекта вы используете свою обёртку.

Часть 7. FAQ
Почему именно '\0', а не длина в начале?
Историческое наследие BCPL → C. Модель дешёва по памяти и проста для ранних систем, но дорога по безопасности и удобству.

Сколько памяти занимает "Hi"?
3 байта: 'H', 'i', '\0'.

Можно ли узнать длину через sizeof?
Только для массива с известным размером на этапе компиляции:

char a[] = "Hi";
size_t len = sizeof(a) - 1;   /* 2 */
Для char * — нет.

Чем '\0' отличается от '0'?
'\0' — нулевой байт (конец строки). '0' — символ нуля (код 48).

Что такое wchar_t, char16_t, char32_t?
Отдельные типы для широких строк. С обычными char * API они не смешиваются без явной конвертации.

Можно ли хранить бинарные данные в char *?
Как буфер байтов — да. Как null-terminated строку — нет, если внутри есть нулевые байты.

Почему gets убрали из стандарта?
Невозможно безопасно использовать: не проверяет размер буфера. Используйте fgets.

Заключение
Текст для выступления

Подведём итог. C-строка — это не объект и не высокоуровневая абстракция. Это договорённость: «здесь лежат байты, и где-то дальше будет ноль».

Новичку важно запомнить три вещи: терминатор '\0', размер буфера и функции из <string.h>.

Опытному — три другие: ownership, lifetime и границы, плюс честный вопрос: нужна ли в этой задаче вообще null-terminated модель.

C дал миру простейший интерфейс для строк. И индустрия потратила годы на то, чтобы научиться не падать на этом простейшем интерфейсе. Если вы понимаете, как это устроено, вы понимаете огромный пласт системного кода — от argv до HTTP-заголовков.
